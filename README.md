# writing-quality_diy — 中文写作质感增强

> 对标 Claude Fable 5 的**文学性输出**风格。通过风格约束 + 多轮自查，消除「AI 味」，实现自然流畅的专业中文写作。

---

## 为什么需要这个技能

DeepSeek V4 Pro 的中文写作能力在开源模型中领先，但与 Fable 5 的文学质感相比仍有差距。典型问题：

- **句式机械**：连续 3 句以上同一结构（"XX 是 YY。XX 可以 ZZ。XX 实现了 WW。"）
- **套话泛滥**：综上所述、值得注意的是、不可否认、从某种意义来说……
- **翻译腔**：不仅……而且……、一方面……另一方面……、极大地提升了……
- **节奏单调**：每段长度完全一致，读起来像说明书而非文章

本技能用**硬约束 + 自查清单**两轮迭代来解决这些问题。

## 何时使用

| ✅ 应该用 | ❌ 不应该用 |
|----------|-----------|
| 报告 / 方案 / 白皮书润色 | 纯信息查询 |
| 对外发布的文档 | 代码和命令行输出 |
| 感觉输出「太AI」的时候 | 翻译任务（用 cn2en_med_diy） |
| 从零撰写需要质感的文本 | 学术论文（保持学术规范优先） |

## 工作流

```
原始文本
    │
    ▼
第一轮：风格评估 + 初稿生成
    ├── 判断原文语体、句长、节奏、用词
    ├── 套用核心写作约束
    └── 产出第一版
    │
    ▼
第二轮：自查优化
    ├── 逐项检查 8 条清单
    ├── 标注并修正问题
    └── 产出最终版
```

## 核心写作约束（严格执行）

1. **句式**：长短句交错，禁止连续 3 句以上相同结构
2. **节奏**：每段开头用短句建立锚点 → 中间展开 → 结尾回扣主题
3. **生动性**：每 500 字至少 1 个具体例子或类比
4. **中国语境**：使用中文读者熟悉的典故、现象，避免翻译腔
5. **禁用词**：综上所述、值得注意的是、不可否认、从某种意义来说、极大地、众所周知
6. **禁用句式**：不仅……而且……、一方面……另一方面……
7. **段落长度**：每段 3-5 句，不超过 150 字
8. **标题规范**：用动词引领，避免名词堆砌

## 使用示例

**输入（AI味浓）**：
> AI 大模型正在改变软件开发的方式。不仅可以提升开发效率，而且可以降低技术门槛。从某种意义来说，这极大地改变了程序员的工作方式。值得注意的是，这一趋势不可否认地会持续下去。

**输出（润色后）**：
> 一个模型吃掉百万行代码，吐出一个可运行的工程——Fable 5 上周干的就是这件事。Stripe 把五千万行 Ruby 交给它，一天跑完。三个月前，同样的活需要两个工程师干一年。这不是效率的问题，是范式塌了——软件开发从「写代码」变成「描述需求」，从「调试」变成「审查」。

## 自查清单

```
□ 有没有连续 3 句以上都是相同句式？
□ 有没有「综上所述」「值得注意的是」等套话？
□ 每个段落是否都控制在 150 字以内？
□ 每 500 字有没有至少一个具体例子？
□ 开头有没有用一个有力的短句抓住注意力？
□ 结尾有没有回扣主题而不是重复？
□ 读起来像人写的吗？（闭眼默读一遍）
□ 有没有翻译腔？（尝试翻译成英文，再译回来，看看是否僵硬）
```
